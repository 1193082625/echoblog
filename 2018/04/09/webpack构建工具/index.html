  要完成前端工程化，少不了工程化工具，requireJS与grunt的出现，改变了业界前端代码的编写习惯，同时也是推动前端工程化的一个基础。
  grunt是一款javascript构建工具，主要完成编译、压缩、合并等一系列工作，后续又出了yeoman、Gulp、webpack等构建工具
  webpack具有Grunt、Gulp对于静态资源自动化构建的能力，但更重要的是，Webpack弥补了requireJS在模块化方面的缺陷，同时兼容AMD与CMD的模块加载规范，具有更强大的JS模块化的功能

自动化构建工具的两种模式

  • 开发模式
    开发模式比较简单，主要就是监听文件变化，自动进行打包、合并的操作
  • 生产模式
    生产模式主要增加了文件压缩、md5名修改、替换html等操作

CDN

  CDN是构建在网络之上的内容分发网络，依靠部署在各地的边缘服务器，通过中心平台的负载平衡、内容分发、调度等功能模块，使用户就近获取所需内容，降低网络拥塞，提高用户访问响应速度和命中率。CDN的关键技术主要有内容存储和分发技术
  CDN网络是在用户和服务器之间增加Cache层，如何将用户的请求引导到Cache上获得服务器的数据，主要是通过接管DNS实现

CommonJS规范

  根据CommonJS规范，每一个文件就是一个模块，其内部定义的变量是属于这个模块的，不会对外暴露（即不会污染全局变量）
  该规范最初是用在服务器端的node的（Nodejs是基于CommonJS规范编写的），前端的webpack也是对CommonJS原声支持的
  CommonJS的核心思想就是通过require方法来同步加载所需要依赖的其他模块，然后通过exports或module.exports来导出需要暴露的接口
// index.js
var moduleDemo = require(‘module.js’);
moduleDemo.aa(‘hello’);

// module.js
module.exports = {
    aa: function(str){
        console.log(str);
    }
}

  浏览器不兼容CommonJS，原因是浏览器缺少module、exports、require、global四个环境变量。如要使用需要工具转换
  CommonJS采用同步加载不同模块文件，适用于服务器端。因为模块文件都存放在服务器的各个硬盘上行，读取加载时间快，适合服务器，不适合浏览器

AMD/CMD

由于CommonJS不适用于浏览器端，浏览器更希望能够实现异步加载的方式。所以衍生出以下两种流派，一种是适用于浏览器的异步加载(AMD)，一种是基于CommonJS的完善(CMD)：

  • AMD
        AMD规范是异步加载模块，允许指定回调函数。等模块异步加载完成后即可调用回调函数，代表是require.js。require.js出来之后AMD规范才被大众熟知
        AMD的核心思想是通过define来定义一个模块，然后使用require来加载一个模块
    // main.js
    require([‘jquery’, ‘math’], function($, math){
    内容
    })
    // math.js
    define(function(){
    内容
    })

    // myLib.js

    return {

Webpack

  在Webpack当中，所有资源都是模块，模块都需要通过AMD或者CMD规范加载，就像CSS样式文件，不再在html中以\<link>标签加载。任何静态资源都可以视作模块，然后模块之间也可以相互依赖，通过webpack对模块进行处理后，可以打包成想要的静态资源

webpack支持的功能特性：

  • 支持CommonJS和AMD模块，基本可以无痛迁移旧项目
  • 支持模块加载器和插件机制，可对模块灵活定制。
  • 可以通过配置，打包成多个文件，有效利用浏览器的缓存功能提升性能
  • 内置有source map,即使打包在一起依旧方便调试
  • 样式文件和图片等静态资源也可视为模块进行打包

webpack配置

1、npm install webpack -g
2、npm init
